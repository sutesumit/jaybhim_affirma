# Telegram Notification Copy Improvements

## Problems Identified

### Problem 1: Visitor notification has no logged-in user info
Current output:
```
🎨 art.sumitsute.com | Art Site
👤 ✨ new
📍 Vijayawada, Andhra Pradesh, IN
💻 Windows
🌐 49.204.148.221
🔗 direct
```
There's no way to tell if the visitor is logged in. If they are, you can't identify who they are.

**Root cause**: `/api/visit/route.ts` never calls `AuthManager.getAuthenticatedUser()`. It only tracks IP, geo, device, and referrer.

### Problem 2: Login notification shows empty contact `()`
Current output:
```
✅ Sute9 () logged in
```

**Root cause**: Two sub-issues:
1. `AuthOtpVerifiedPayload` has a `phone: string` field but the value could be email or phone — confusing naming.
2. **Empty string bug**: In `verify-otp/route.ts`, `contact = user.phone ?? user.email ?? "unknown"` uses nullish coalescing (`??`). If Supabase returns `user.phone = ""` (empty string), `??` doesn't catch it — it passes through, and `maskContact("")` returns `""`.

### Problem 3: Anonymous comments don't indicate anonymity
Comments have an `isAnonymous` flag. When someone posts anonymously, the notification still shows their real name and contact. The owner should still see the real identity (it's an admin notification), but it should indicate the comment was posted anonymously.

### Problem 4: Long story/comment text could exceed Telegram's 4096-char limit
Telegram messages cap at **4096 characters**. Stories can be **2500 chars** and comments **2000 chars**. Edit notifications show both old + new text, which could total **5000+ chars**. Current truncation (80/40 chars) is aggressive but safe — we should increase it slightly while adding a safety clamp to ensure we never exceed the limit.

### Problem 5: All action notifications missing contact/IP
Current output:
```
❤️ Sute9 liked /fathers-and-figures total 4
📖 Sute9 submitted story "Test Story."
💬 Sute9 commented on /fathers-and-figures "Test"
```
No masked phone/email, no IP shown.

**Root cause**: Same empty-string bug. All routes use `user.phone ?? user.email ?? null`. If `user.phone` is `""`, it falls through without hitting `user.email`. Should use `||` instead of `??` to catch empty strings.

## Fix Plan

### Step 1: Fix empty-string contact bug across all routes

Replace `??` with `||` for contact extraction in **all 7 route files**. This ensures empty strings are treated as falsy:

```typescript
// Before (broken with empty strings)
const contact = user.phone ?? user.email ?? null;

// After (handles empty strings correctly)
const contact = user.phone || user.email || null;
```

**Files to update:**
- `src/app/api/auth/verify-otp/route.ts` — line 136
- `src/app/api/auth/logout/route.ts` — line 11
- `src/app/api/likes/route.ts` — line 172
- `src/app/api/comments/route.ts` — line 111
- `src/app/api/comments/[id]/route.ts` — lines 153, 271
- `src/app/api/father-son-stories/route.ts` — line 92
- `src/app/api/father-son-stories/[storyId]/route.ts` — lines 95, 180

### Step 2: Add logged-in user info to visitor notifications

#### 2a. Update `VisitorNotificationPayload` in `types.ts`
Add optional auth fields:
```typescript
export type VisitorNotificationPayload = {
  city?: string;
  country?: string;
  region?: string;
  ip?: string;
  deviceType?: string;
  isReturning?: boolean;
  visitCount?: number;
  timestamp?: string;
  // New auth fields:
  userName?: string;
  contact?: string | null;
};
```

#### 2b. Resolve auth in `/api/visit/route.ts`
Call `AuthManager.getAuthenticatedUser()` and pass user info to the service:
```typescript
import { AuthManager } from "@/lib/auth/auth-manager";

// In the POST handler, before calling visitService.trackVisit():
const authUser = await AuthManager.getAuthenticatedUser();
const userName = authUser?.display_name ?? undefined;
const contact = authUser?.phone || authUser?.email || undefined;
```

Pass `userName` and `contact` through to the notification payload in `service.ts`.

#### 2c. Update visit service to forward auth info
In `src/lib/visit/service.ts`, accept `userName`/`contact` in the `trackVisit` call and include them in the `notifyVisitor` payload.

#### 2d. Update `formatVisitorNotification` in `formatters.ts`
Add a user line to the notification when logged in:
```
🎨 art.sumitsute.com | Art Site
👤 ✨ new (1) — Sute9 (su***@gmail.com) 🔐
📍 Vijayawada, Andhra Pradesh, IN
💻 Windows
🌐 49.204.148.221
🔗 direct
```

If not logged in, the notification stays the same as before (no 🔐 line).

### Step 3: Add anonymous flag to comment notifications

#### 3a. Add anonymous fields to comment notification types
In `types.ts`:
- Add `isAnonymous?: boolean` to `CommentNotificationPayload` (create comment)
- Add `wasAnonymous?: boolean` and `isAnonymous?: boolean` to `CommentEditNotificationPayload` (edit comment — shows old and new anonymous state)
- `CommentDeleteNotificationPayload` doesn't need it — the comment's current state is implicit

#### 3b. Pass anonymous state from routes
- `comments/route.ts` — pass `isAnonymous: !!isAnonymous` in the notification payload
- `comments/[id]/route.ts` (edit) — pass `wasAnonymous: !!existingComment.is_anonymous` and `isAnonymous: typeof isAnonymous === "boolean" ? !!isAnonymous : !!existingComment.is_anonymous` (the new value, or unchanged if not provided)
- `comments/[id]/route.ts` (delete) — no change needed

#### 3c. Show anonymous state in formatters
- **Comment create**: append `[anonymous]` badge when `isAnonymous` is true
- **Comment edit**: show flag transition when it changed:
  - `[public → anonymous]` — changed to anonymous
  - `[anonymous → public]` — changed to public
  - `[anonymous]` — was and stays anonymous
  - (no badge) — was and stays public
- **Comment delete**: no badge (owner can check the comment's state separately)

Example outputs:
```
💬 Sute9 (su***@gmail.com) [anonymous] commented on /page "Test"
💬 Sute9 (su***@gmail.com) [public → anonymous] edited comment on /page "Test" → "Test 2"
```

### Step 4: Smart truncation for long text

#### 4a. Add a `truncateWithCount` helper to `formatters.ts`
Instead of silently truncating, show a character count so the owner knows how long the full text is:
```typescript
function truncateWithCount(text: string, maxLen: number): string {
  if (text.length <= maxLen) return text;
  const truncated = text.slice(0, maxLen - 3) + "...";
  return `${truncated} (${text.length} chars)`;
}
```

#### 4b. Increase truncation limits with safety margin
- Full text (story submit, comment post): **200 chars** (was 80)
- Edit text (old/new for edits, deletes): **120 chars** each (was 40)
- Add a final `ensureUnderLimit(message, 4000)` clamp that cuts from the text portion if the total message exceeds ~4000 chars (leaving 96 chars buffer for header/IP/etc)

Example output for a 2500-char story:
```
📖 Sute9 (su***@gmail.com) submitted story
"Lorem ipsum dolor sit amet, consectetur adipiscing elit. Sed do eiusmod tempor incididunt ut labore et dolore magna aliqua. Ut enim ad minim veniam, quis nostrud exercitation ullamco laboris... (2467 chars)"
```

### Step 5: Fix login notification formatter

#### 5a. Rename `AuthOtpVerifiedPayload.phone` to `contact`
```typescript
export type AuthOtpVerifiedPayload = {
  contact: string;       // was: phone: string
  isNewUser: boolean;
  userName?: string;
  ip?: string | null;
};
```

#### 5b. Update `verify-otp/route.ts` to pass `contact` instead of `phone`
```typescript
void telegramNotifier.notifyOtpVerified({
  contact,          // was: phone: contact
  isNewUser,
  userName: displayNameToUse,
  ip,
})
```

#### 5c. Update `formatOtpVerifiedNotification` to handle empty contact gracefully
```typescript
// Before
✅ Sute9 (<code>${maskedPhone}</code>) logged in

// After — show contact inline with name, or just name if no contact
✅ Sute9 (su***@gmail.com) logged in       // with contact
✅ Sute9 logged in                          // without contact
```

Use `formatUserWithContact` (same as other notifications) for consistency.
