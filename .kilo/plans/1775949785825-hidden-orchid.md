# Telegram Notification System — Implementation Plan

## Overview

Adapt the `TELEGRAM_NOTIFICATION_BLUEPRINT.md` for the **jaybhim_affirma** project (art/documentary portfolio). The system sends one-way Telegram notifications to the owner's DM for key user interactions.

### Scope

- **Notification types**: Visitor tracking, Likes, Comments, Story submissions
- **Direction**: Owner DM only (no channel broadcast)
- **Bot interaction**: One-way notifications only (no webhook, no commands)

---

## Notification Types Summary

| Type | Channel | Trigger | Format |
|------|---------|---------|--------|
| Visitor | Owner DM | Page visit with IP geolocation | `👤 new/returning (N) 📍 City, Country 💻 Device 🌐 IP 🔗 referrer` |
| Like | Owner DM | User likes a page | `❤️ ip X liked | page_path | total 42` |
| Comment | Owner DM | User posts a comment | `💬 ip X commented on | page_path | "comment text..."` |
| Story | Owner DM | User submits a father-son story | `📖 ip X submitted story | story_title` |

### Phase 2: Core Infrastructure (new files)

#### 2.1 Bot Singleton — `src/lib/telegram/bot.ts`
- Create minimal bot singleton using grammy
- No command handlers (one-way only)
- Lazy `init()` with `TELEGRAM_BOT_TOKEN` check
- Export `initBot()` and `sendMessage()` helpers

#### 2.2 Auth Middleware — `src/lib/telegram/middleware/auth.ts`
- `getAllowedUserIds()` — parse `TELEGRAM_ALLOWED_USER_IDS` env var

### Phase 3: Notification Types & Formatting (new files)

#### 3.1 Types — `src/lib/notifications/types.ts`
Define payload interfaces:
- `VisitorNotificationPayload` — ip, city, country, region, deviceType, isReturning, visitCount, timestamp
- `LikeNotificationPayload` — pagePath, likeCount, ip
- `CommentNotificationPayload` — pagePath, commentText (truncated), userName, ip
- `StoryNotificationPayload` — storyTitle, storyId, ip
- `TelegramNotifier` interface with methods: `notifyVisitor`, `notifyLike`, `notifyComment`, `notifyStory`
- `noopTelegramNotifier` — no-op fallback for dev/test

#### 3.2 Formatters — `src/lib/notifications/formatters.ts`
- `escapeHtml()` — sanitize user content for Telegram HTML
- `formatVisitorNotification()` — reuse blueprint format with project header
- `formatLikeNotification()` — page path + total likes
- `formatCommentNotification()` — page path + truncated comment text
- `formatStoryNotification()` — story title + link

**Project constants:**
```typescript
const SITE_URL = "https://art.sumitsute.com"; // or actual domain
const PROJECT_HEADER = "🎨 art.sumitsute.com";
```

#### 3.3 Notifier Implementation — `src/lib/notifications/telegram-notifier.ts`
- `TelegramBotNotifier` class implementing `TelegramNotifier`
- Uses lazy `await import("@/lib/telegram/bot")` in `sendMessage()` to break circular deps
- All methods: get owner chat ID, format message, send via bot API
- Export singleton `telegramNotifier`

### Phase 4: Visitor Tracking (new files)

#### 4.1 Visit Types — `src/lib/visit/types.ts`
- `VisitRequestPayload` — ip, referrer
- `VisitRepository` interface — `upsertVisitorState()`, `getMostRecentVisitor()`, `countUniqueVisitors()`

#### 4.2 Visit Repository — `src/lib/visit/repository.ts`
- Supabase implementation of `VisitRepository`
- Uses `supabase.rpc('upsert_visit_state', ...)` for atomic upsert

#### 4.3 Visit Service — `src/lib/visit/service.ts`
- DI pattern: `createVisitService({ repository, notifier, now })`
- `trackVisit(payload, userAgent)` — parse device, upsert visitor, fire-and-forget notification
- `parseDeviceType()` utility (reuse from blueprint)
- Export `visitService` singleton wired with real deps

#### 4.4 Visit API Route — `src/app/api/visit/route.ts`
- POST handler: parse body (ip, referrer), extract user-agent, call `visitService.trackVisit()`
- Return visitor summary (location, count)

#### 4.5 Client Hook — `src/hooks/useAnalytics.ts`
- `useAnalytics()` — runs once per mount, fetches IP from `ipapi.co/json`, POSTs to `/api/visit`
- Wire into `src/app/layout.tsx` (add to root layout)

### Phase 6: Database (Supabase)

#### 6.1 Visitors table migration
```sql
CREATE TABLE visitors (
  id BIGSERIAL PRIMARY KEY,
  ip inet NOT NULL,
  city text,
  region text,
  country text,
  visit_count integer DEFAULT 1,
  last_visit_time timestamptz DEFAULT now(),
  created_at timestamptz DEFAULT now()
);
CREATE UNIQUE INDEX idx_visitors_ip ON visitors(ip);

CREATE OR REPLACE FUNCTION upsert_visit_state(p_ip inet)
RETURNS TABLE(ip inet, city text, region text, country text, visit_count integer, last_visit_time timestamptz)
LANGUAGE plpgsql AS $$
BEGIN
  INSERT INTO visitors (ip, visit_count)
  VALUES (p_ip, 1)
  ON CONFLICT (ip) DO UPDATE SET
    visit_count = visitors.visit_count + 1,
    last_visit_time = now()
  RETURNING visitors.ip, visitors.city, visitors.region, visitors.country, visitors.visit_count, visitors.last_visit_time;
END;
$$;
```

Run via Supabase dashboard or CLI.

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| No webhook route | One-way only — bot never receives commands |
| No channel broadcast | Owner DM only — simpler setup |
| No broadcast secret | Not needed without channel |
| DI pattern for visit service | Testable, swappable notifier |
| Fire-and-forget with `void` | Notifications must not block API responses |
| Lazy import in sendMessage | Breaks circular dependency between notifier ↔ bot |
| No bot commands | No interactive bot features needed |
