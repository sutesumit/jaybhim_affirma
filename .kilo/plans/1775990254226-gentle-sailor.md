# Plan: Enable RLS on Unprotected Tables

## Problem

3 tables have RLS **disabled**, exposing them to unrestricted access from anyone with the public anon key:

| Table | RLS | Rows | Risk Level | Exposure |
|---|---|---|---|---|
| `father_son_stories` | ❌ OFF | 25 | **HIGH** | Anyone can read (incl. deleted), insert fake stories, modify/delete others' stories |
| `user_interaction_counters` | ❌ OFF | 11 | **MEDIUM** | Anyone can inflate counts, read all user interaction data |
| `visitors` | ❌ OFF | 7 | **MEDIUM** | Anyone can harvest all visitor IPs, locations, visit counts |

**Key architecture fact:** The app uses the **anon key everywhere** (no service_role key). Server API routes use `getServerSupabase()` which injects the user's JWT when authenticated, or falls back to anon. This means RLS policies will correctly gate all access.

---

## Phase 1: `father_son_stories` (HIGH risk — direct .from() access)

### Current Access Patterns (all via `getServerSupabase`):
- **GET** `/api/father-son-stories` — Public read, filters `deleted_at IS NULL`, optional `userId` filter
- **POST** `/api/father-son-stories` — Authenticated only, inserts with `user_id` from auth
- **PATCH** `/api/father-son-stories/[id]` — Authenticated only, verifies ownership, updates active story
- **DELETE** `/api/father-son-stories/[id]` — Authenticated only, verifies ownership, soft-deletes

### Migration: Enable RLS + 4 Policies

```sql
ALTER TABLE father_son_stories ENABLE ROW LEVEL SECURITY;

-- 1. Public read active stories (covers GET endpoint for anonymous visitors)
CREATE POLICY "Public read active stories" ON father_son_stories
  FOR SELECT TO public USING (deleted_at IS NULL);

-- 2. Owners can see their own deleted stories (for potential future use)
CREATE POLICY "Owners can read their deleted stories" ON father_son_stories
  FOR SELECT TO authenticated USING (auth.uid() = user_id);

-- 3. Authenticated users can insert stories (must be their own)
CREATE POLICY "Authenticated users can insert stories" ON father_son_stories
  FOR INSERT TO authenticated WITH CHECK (auth.uid() = user_id);

-- 4. Users can update own active stories (covers edit + soft-delete)
CREATE POLICY "Users can update own stories" ON father_son_stories
  FOR UPDATE TO authenticated
  USING (auth.uid() = user_id AND deleted_at IS NULL)
  WITH CHECK (auth.uid() = user_id);
```

### Compatibility Analysis
| Endpoint | API Route Behavior | RLS Policy | Works? |
|---|---|---|---|
| GET (public) | `getServerSupabase` (anon), `.is("deleted_at", null)` | Public SELECT `deleted_at IS NULL` | ✅ |
| GET (auth) | Same as above | Auth SELECT `auth.uid() = user_id` (sees own deleted) | ✅ |
| POST | Auth-only guard, inserts with `user_id` from auth | INSERT `auth.uid() = user_id` | ✅ |
| PATCH | Auth-only, fetches first (checks ownership + active), then updates | UPDATE `auth.uid() = user_id AND deleted_at IS NULL` | ✅ |
| DELETE | Auth-only, fetches first (checks ownership + active), then soft-deletes | UPDATE `auth.uid() = user_id AND deleted_at IS NULL` → sets `deleted_at` | ✅ |

**No code changes needed.** Triggers bypass RLS (run as table owner).

## Phase 2: `user_interaction_counters` (MEDIUM risk — trigger-written)

### Current Access:
- **Written by** triggers only (on `likes`, `comments`, `father_son_stories`) — bypass RLS ✅
- **Read by** `get_top_user` RPC (NOT SECURITY DEFINER) — **will break** with RLS
- **Read by** `get_interaction_analytics` RPC (IS SECURITY DEFINER) — bypasses RLS ✅
- **No direct `.from()` access** from app code

### Migration: Enable RLS + Make `get_top_user` SECURITY DEFINER

```sql
ALTER TABLE user_interaction_counters ENABLE ROW LEVEL SECURITY;
-- No policies needed. All writes via triggers (bypass RLS).
-- All reads via SECURITY DEFINER RPCs (bypass RLS).
```

Update `get_top_user` to add `SECURITY DEFINER`:

```sql
CREATE OR REPLACE FUNCTION public.get_top_user(current_user_id uuid)
RETURNS TABLE(display_name text, like_count bigint, comment_count bigint, father_son_stories_count bigint, total bigint, updated_at timestamp with time zone, isme boolean)
LANGUAGE plpgsql
STABLE SECURITY DEFINER
SET search_path = public
AS $function$
BEGIN
  RETURN QUERY
  SELECT
    p.display_name,
    uic.like_count,
    uic.comment_count,
    uic.father_son_stories_count,
    (uic.like_count + uic.comment_count + uic.father_son_stories_count) AS total,
    uic.updated_at,
    (p.id = current_user_id) IS TRUE AS isMe
  FROM public.user_interaction_counters uic
  JOIN public.profiles p ON p.id = uic.user_id
  WHERE (uic.like_count + uic.comment_count + uic.father_son_stories_count) > 0
  ORDER BY total DESC, uic.updated_at DESC
  LIMIT 1;
END;
$function$;
```

**No app code changes needed.**

## Phase 3: `visitors` (MEDIUM risk — IP data exposure)

### Current Access (via bare `supabase` anon client, NO JWT):
- `upsert_visit_state` RPC — INSERT/UPDATE + SELECT (NOT SECURITY DEFINER)
- `.from("visitors").select()` — get most recent visitor for a specific IP
- `.from("visitors").select("*", { count: "exact", head: true })` — count all visitors

### Strategy: SECURITY DEFINER RPCs + Remove Direct Table Access

The `upsert_visit_state` RPC already returns all needed data (ip, city, country, region, visitCount, lastVisitTime). The separate `getMostRecentVisitor` call is **redundant** — same data already available from the RPC return.

#### Step 1: Migration — Enable RLS + SECURITY DEFINER RPCs

```sql
-- Lock down visitors table
ALTER TABLE visitors ENABLE ROW LEVEL SECURITY;
-- No policies = fully locked down (IPs protected)

-- Make upsert_visit_state SECURITY DEFINER (bypasses RLS)
CREATE OR REPLACE FUNCTION public.upsert_visit_state(target_ip inet)
RETURNS TABLE(r_ip inet, r_city text, r_region text, r_country text, r_visit_count integer, r_last_visit_time timestamp with time zone)
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $function$
BEGIN
  INSERT INTO visitors (ip, visit_count)
  VALUES (target_ip, 1)
  ON CONFLICT (ip) DO UPDATE SET
    visit_count = visitors.visit_count + 1,
    last_visit_time = now();

  RETURN QUERY
  SELECT v.ip, v.city, v.region, v.country, v.visit_count, v.last_visit_time
  FROM visitors v
  WHERE v.ip = target_ip;
END;
$function$;

-- New RPC for visitor count (SECURITY DEFINER)
CREATE OR REPLACE FUNCTION public.count_unique_visitors()
RETURNS bigint
LANGUAGE sql
SECURITY DEFINER
SET search_path = public
AS $$
  SELECT count(*)::bigint FROM visitors;
$$;
```

#### Step 2: Refactor `src/lib/visit/repository.ts`

- Remove `getMostRecentVisitor()` method (redundant — data from RPC return)
- Change `countUniqueVisitors()` to use `supabase.rpc("count_unique_visitors")` instead of direct `.from("visitors").select()`

```ts
// Before (3 methods):
upsertVisitorState(payload)      // RPC call
getMostRecentVisitor(ip)          // direct .from() SELECT — REMOVED
countUniqueVisitors()             // direct .from() SELECT — CHANGED TO RPC

// After (2 methods):
upsertVisitorState(payload)      // RPC call (unchanged)
countUniqueVisitors()             // NEW: supabase.rpc("count_unique_visitors")
```

#### Step 3: Refactor `src/lib/visit/service.ts`

Use `visitorState` (already returned from `upsertVisitorState`) for location/time data instead of the redundant `getMostRecentVisitor` call:

```ts
// Before:
const [lastVisitor, uniqueCount] = await Promise.all([
  body.ip ? repository.getMostRecentVisitor(body.ip) : Promise.resolve(null),
  repository.countUniqueVisitors(),
]);
return {
  lastVisitorLocation: lastVisitor ? formatVisitorLocation(...) : null,
  lastVisitTime: lastVisitor?.lastVisitTime ?? null,
  visitorCount: uniqueCount,
};

// After:
const uniqueCount = await repository.countUniqueVisitors();
return {
  lastVisitorLocation: visitorState
    ? formatVisitorLocation(visitorState.city, visitorState.country) : null,
  lastVisitTime: visitorState?.lastVisitTime ?? null,
  visitorCount: uniqueCount,
};
```

#### Step 4: Update `src/lib/visit/types.ts`

Remove `getMostRecentVisitor` from the `VisitRepository` interface.

| File | Change | Risk |
|---|---|---|
| **Migration SQL** | Enable RLS on 3 tables + policies on `father_son_stories` + make 3 RPCs SECURITY DEFINER | DB-only |

| `src/lib/visit/service.ts` | Use `visitorState` for return data instead of redundant query | Low |

### What stays working:
- ✅ All `father_son_stories` CRUD (public read, auth write, owner-only edit/delete)
- ✅ All trigger-based counter updates on `user_interaction_counters`
- ✅ Leaderboard via `get_top_user` (now SECURITY DEFINER)
- ✅ Analytics via `get_interaction_analytics` (already SECURITY DEFINER)
- ✅ Visit tracking via `upsert_visit_state` (now SECURITY DEFINER)
- ✅ Visitor count via new `count_unique_visitors` RPC

### What's now protected:
- 🔒 `father_son_stories` — anon can only read active stories; auth users can only modify their own
- 🔒 `user_interaction_counters` — fully locked down (only triggers + SECURITY DEFINER RPCs)
- 🔒 `visitors` — fully locked down (IPs not readable; only SECURITY DEFINER RPCs)
